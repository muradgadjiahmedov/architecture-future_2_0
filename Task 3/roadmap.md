# Task3 — Роадмап изменений (12 месяцев)

## Этап 1. Архитектурная стабилизация (0–2 месяца)
**Цель бизнеса:** согласованное видение и быстрые победы  

**Ключевые активности:**
- Утверждение доменной модели и целевой архитектуры
- Запуск cloud data lake (raw)
- Настройка IAM и базовой security-модели
- Выделение медицинского контура (restricted)

**Результаты:**
- Архитектурное решение согласовано
- Снижены регуляторные риски

**Команды:** Architecture, Security, Data Platform  
**Ресурсы:** облако, архитекторы, ИБ

---

## Этап 2. Запуск витрины данных (2–6 месяцев)
**Цель бизнеса:** ускорение принятия решений  

**Ключевые активности:**
- Lakehouse (refined слой)
- Доменные аналитические витрины (Clinic, Fintech)
- Semantic layer и единые KPI
- Первый MVP портала самообслуживания

**Результаты:**
- Отчёты формируются за минуты
- Бизнес сам собирает отчёты

**Команды:** Data Platform, Analytics, Business  
**Ресурсы:** data engineers, BI, cloud compute

---

## Этап 3. Независимость доменов (6–9 месяцев)
**Цель бизнеса:** масштабирование и гибкость  

**Ключевые активности:**
- Event Bus как основной способ интеграции
- Data Products по Data Mesh
- Governance: каталог, lineage, DQ

**Результаты:**
- Новые бизнесы подключаются без изменений DWH
- Повышено доверие к данным

**Команды:** Domain teams, Platform  
**Ресурсы:** streaming, governance tooling

---

## Этап 4. Оптимизация и вывод legacy (9–12 месяцев)
**Цель бизнеса:** снижение затрат и техдолга  

**Ключевые активности:**
- Перевод BI на semantic layer
- Вывод логики из legacy DWH
- Замена PowerBuilder в приоритетных доменах

**Результаты:**
- Снижение стоимости поддержки
- Устойчивость архитектуры

**Команды:** Platform, Product teams  
**Ресурсы:** frontend, data migration
